# Async Report Generator

Сервис для асинхронной генерации тяжелых отчетов в форматах Excel, PDF и CSV с доставкой файлов через S3-совместимое хранилище (MinIO).

## Основной функционал

### 1. Управление заявками на отчет (Report Task)
- **Создание задачи**:
  - Пользователь отправляет запрос с параметрами фильтрации, названием отчета и желаемым форматом.
  - Система регистрирует задачу в БД со статусом `pending`.
  - В ответ возвращается уникальный идентификатор задачи (`task_id`).
- **Проверка статуса**:
  - По `task_id` можно получить текущий статус, прогресс (опционально) и сообщение об ошибке, если генерация провалилась.
- **Получение результата**:
  - Для задач в статусе `completed` сервис генерирует одноразовую pre-signed ссылку на скачивание файла из S3 (действует ограниченное время).
- **Список задач пользователя**:
  - Возможность получить историю всех сгенерированных отчетов с пагинацией.

### 2. Процесс генерации (Асинхронный пайплайн)
- **Постановка в очередь**:
  - При создании задачи, Use Case отправляет сообщение в RabbitMQ (Exchange: `report.tasks`, Routing Key: `generate`).
- **Обработка воркером (Celery)**:
  - Воркер забирает задачу, меняет статус на `processing`.
  - Выполняется сбор данных согласно переданным `query_params` (например, запрос к аналитической БД или внешнему API).
  - Данные агрегируются и преобразуются в файл нужного формата (`.xlsx`, `.pdf`, `.csv`).
  - Готовый файл загружается в бакет MinIO.
  - Статус задачи обновляется на `completed`, в БД записывается путь к файлу в S3.
- **Обработка ошибок**:
  - Если на любом этапе произошел сбой, статус меняется на `failed`, а в поле `error_message` пишется трассировка/причина.

### 3. Форматы и данные
- **Поддерживаемые форматы**:
  - Excel (`.xlsx`) — дефолтный.
  - CSV (`.csv`).
  - PDF (`.pdf`) — генерация из HTML-шаблона.
- **Источники данных (заглушка для MVP)**:
  - В первой версии источником будет моковый генератор случайных данных (имитация выгрузки из БД).
  - В будущем — подключение к реальной Postgres (через SQLAlchemy) или внешнему REST API.

### 4. Мониторинг и Observability
- **Prometheus Metrics**:
  - `report_requests_total` (по статусам).
  - `report_generation_duration_seconds` (гистограмма).
  - `celery_queue_length` (длина очереди RabbitMQ).
  - `s3_upload_speed_bytes` / `s3_errors_total`.
- **Grafana Dashboard**:
  - График RPS создания задач.
  - График загрузки очереди и количества активных воркеров.
  - Топ-5 самых долгих генераций.
- **Логирование (ELK)**:
  - Структурированные JSON-логи от FastAPI и Celery.
  - `correlation_id` (trace_id), передаваемый от запроса пользователя до воркера для удобного дебага.

### 5. Хранение данных
- **PostgreSQL**:
  - Таблица `reports` (id, user_id, name, format, query_params, status, created_at, completed_at, file_key, error_message).
- **Redis**:
  - Кеширование статусов часто запрашиваемых задач (TTL 60s).
  - Блокировки для предотвращения повторной обработки одного и того же task_id (если вдруг очередь задвоит сообщение).
- **MinIO (S3)**:
  - Бакет `reports`. Имя объекта: `{user_id}/{year}/{month}/{task_id}.{ext}`.
  - Автоматическая очистка файлов старше 7 дней (через Bucket Lifecycle Policy).

### 6. Безопасность и ограничения
- **Аутентификация**:
  - Упрощенная — заголовок `X-User-Id` (для MVP).
- **Rate Limiting**:
  - Не более 5 запросов на создание отчета в минуту с одного `user_id` (реализовано через Redis).
- **Валидация**:
  - `query_params` проверяются через Pydantic схему на допустимые ключи и типы.


## Структура эндпоинтов API (черновик)

| Метод | Путь                         | Описание                                     |
|-------|------------------------------|----------------------------------------------|
| POST  | `/api/v1/reports`            | Создать заявку на генерацию отчета           |
| GET   | `/api/v1/reports`            | Список отчетов пользователя (пагинация)      |
| GET   | `/api/v1/reports/{task_id}`  | Получить статус и метаданные задачи          |
| GET   | `/api/v1/reports/{task_id}/download` | Получить временную ссылку на скачивание     |

## Сценарий MVP (Что делаю в первую очередь)

1. **Локальное окружение**: `docker-compose.yml` с Postgres, Redis, RabbitMQ, MinIO.
2. **FastAPI + Clean Architecture**: Скелет проекта, `POST /reports` сохраняет в БД.
3. **Celery Worker**: Таска-заглушка, которая ждет 10 секунд, создает фейковый файл и загружает в MinIO.
4. **GET Status & Download**: Возврат статуса и ссылки на файл.
5. **Prometheus**: Подключение метрик к FastAPI и Celery.
